Overview

All functionality of the site is mirrored through the Api. To access POST functionality you must include an authentication token, which is available in the {% if session.logged_in %}settings{% else %}settings{% endif %} page of your account. GET requests can be freely made in most cases for debugging purposes--the server response will echo your request.

Endpoints
localhost:5000/api/auth
localhost:5000/api/city
localhost:5000/api/job
localhost:5000/api/interaction

api/auth

GET

Requests are echoed in a uniform format. Route is the url extension to which the request is send. Token is the user supplied token, which should be placed in the request header. If the token is placed in the request parameters instead of the header, a message is returned informing the user of this fact. Keys are simply the request parameter keys, in the same way Values are parameter values. These attributes are not modified in any way from the originating request and are simply present to allow users to test and debug their functions. Message is a simple response from the server informing the user of any potential issues with their request.

HEADER

Token

PARAMS

Token *

RESPONSE

Route Token Data Keys Values Message

POST

Requests will return a boolean response indicating whether or not the supplied token is valid. Token should be placed in the request header, although validation will still be attempted if placed in the parameters. However, for all other routes the token should be placed in the header, and requests differing from this format will not be accepted.
